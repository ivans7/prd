# Business Logic and Workflow Management Model

Описание модели бизнес-логики и управления рабочими процессами (Workflow) платформы Alloy Navigator.

---

## 1. Архитектурный принцип: разделение платформы и бизнес-логики

Центральный архитектурный принцип платформы — **разделение Platform Capabilities и Business Logic**.

| Слой | Ответственность |
|------|-----------------|
| **Platform Capabilities** | Механизмы управления объектами, автоматизации, связями и обработкой данных |
| **Business Logic** | Пользовательские правила, состояния, переходы и процессы, определяющие поведение объектов |

Платформа **не навязывает** конкретных бизнес-процессов, статусов или жизненных циклов. Она предоставляет инструменты, с помощью которых организация конфигурирует свою собственную логику.

**Что предоставляет платформа (machinery):**
- Определения полей и их настройку
- Классификационные списки и иерархии
- Выполнение действий и триггеров
- Вычисление условий
- Доставку уведомлений
- Управление связями между объектами

**Что определяет организация (logic):**
- Какие статусы существуют и что они означают
- Через какие стадии проходят объекты
- Какие действия доступны и когда
- Какие условия запускают автоматизацию
- Какие уведомления отправляются и кому

## 2. Workflow — ядро модели поведения

### 2.1 Определение

**Workflow** (рабочий процесс) — это полная конфигурация, определяющая поведение класса бизнес-объектов на платформе. Workflow описывает последовательность процедурных шагов бизнес-процесса и применимые правила.

Workflow управляет жизненным циклом объекта от создания до завершения. Жизненный цикл отображается через статусы, которые представляют различные фазы бизнес-процесса.

### 2.2 Структура Workflow

Workflow состоит из двух основных категорий элементов:

```
Workflow
├── Workflow Items (исполняемые единицы)
│   ├── Actions (действия)
│   │   ├── Create Actions (создание объектов)
│   │   ├── Step Actions (шаги жизненного цикла)
│   │   └── Service Actions (интеграционные действия)
│   ├── Triggers (триггеры)
│   │   ├── On Condition (по условию)
│   │   ├── On Created/Modified (при создании/изменении)
│   │   └── On Delete (при удалении)
│   └── Business Policies (бизнес-политики, только для тикетов)
│       ├── Escalation (эскалация)
│       ├── Prioritization (приоритизация)
│       ├── Routing (маршрутизация)
│       └── Additional Automation (доп. автоматизация)
│
└── Workflow Components (переиспользуемые строительные блоки)
    ├── Templates (шаблоны инициализации)
    ├── Forms (формы ввода)
    ├── Functions (функции)
    └── Notifications (уведомления)
        ├── E-mail Notifications
        └── Push Notifications
```

### 2.3 Навигация по конфигурации

Вся конфигурация бизнес-логики организована в Settings App:

```
Workflow and Business Logic > [Module] > [Object Class] > Workflow
```

Например: `Workflow and Business Logic > Service Desk > Incidents > Workflow`

Альтернативный вид — по процессам: `Workflow and Business Logic > Advanced > Processes`

## 3. Workflow Items: Actions (Действия)

Actions — основной механизм взаимодействия пользователей и сервисов с объектами. Это операции, которые модифицируют бизнес-объекты.

### 3.1 Create Actions (действия создания)

Определяют, как создаются и инициализируются новые объекты. Каждый объект должен иметь хотя бы одно Create Action. Может существовать несколько Create Actions для одного класса, каждое с разной логикой инициализации.

**Состав Create Action:**

| Компонент | Назначение |
|-----------|-----------|
| **General** | Имя, заголовок, подсказка, описание, флаг включения |
| **Condition** | Ограничения по роли безопасности + дополнительные критерии доступности |
| **Template** | Назначает начальные значения полям нового объекта |
| **Form** | Собирает данные от пользователя (диалог или мастер) |
| **Programming** | Операции, выполняемые при создании объекта |

**Доступ:** через меню **New** или **File > New**.

**Логика Programming** позволяет при создании объекта:
- Инициализировать вычисляемые поля
- Создавать связанные объекты
- Устанавливать связи
- Вызывать переиспользуемые функции
- Отправлять уведомления
- Вызывать внешние интеграции

### 3.2 Step Actions (шаговые действия)

Выполняют операции над существующими объектами в течение их жизненного цикла:
- Переходы между пользовательскими классификационными значениями (статусами)
- Модификации полей (аннотации, категоризация, приоритизация)
- Изменения связей (привязка связанных объектов, создание задач)

**Конфигурация Step Action** включает те же 5 вкладок, что и Create Action, плюс:
- **Condition** содержит дополнительные фильтры по **Types** и **Statuses** (ограничение действия определёнными фазами)
- **Scope** может быть "Multiple objects" (массовые действия) или "Single object only"

**Управление доступностью:**
- **По статусу:** определяет, при каких статусах действие может быть выполнено
- **По роли:** определяет, какие роли безопасности имеют право выполнять действие

**Доступ:** через **Action Bar**, меню **Actions**, контекстное меню. Недоступные шаги отображаются серыми (конфигурируемо — могут скрываться).

### 3.3 Service Actions (сервисные действия)

Инициируются интеграционными каналами, а не пользователями:

| Тип | Назначение |
|-----|-----------|
| **Create Service Action** | Определяет, как внешние сервисы создают объекты |
| **Update Service Action** | Определяет, как внешние сервисы модифицируют существующие объекты |

**Поддерживаемые сервисы:** Mail Connector, Alloy Discovery Connector, Active Directory Import, Self Service Portal (SSP).

Включают вкладку **Field Mapping** для сопоставления атрибутов источника с полями Alloy Navigator.

## 4. Workflow Items: Triggers (Триггеры)

Triggers — автоматические операции, выполняемые на основании условий или событий. **Не запускаются** на неактивных объектах (Closed, Retired, Inactive).

### 4.1 Типы триггеров

| Тип | Когда срабатывает | Особенности |
|-----|------------------|-------------|
| **On Condition** | Когда существующий объект удовлетворяет условию | Проверяется по расписанию (каждые X минут) или при каждом изменении; результаты записываются в лог |
| **On Created/Modified** | При создании или модификации объекта | Срабатывает **до** сохранения данных в БД; может быть ограничен только Full Form |
| **On Delete** | При удалении объекта | — |

### 4.2 Конфигурация триггера

| Вкладка | Содержание |
|---------|-----------|
| **General** | Имя, описание, включение, поведение срабатывания, расписание |
| **Condition** | Построитель условий для определения момента срабатывания |
| **Programming** | Операции для выполнения (тот же набор, что у Actions) |
| **Processes** | Привязка к рабочим процессам |

### 4.3 Управление повторным срабатыванием

On Condition Triggers поддерживают контроль повторного срабатывания:
- Однократное срабатывание
- Повторное срабатывание с ограничением частоты
- Условие остановки срабатывания

Это предотвращает бесконечные циклы и чрезмерные повторения.

### 4.4 Типичные сценарии использования триггеров

- **Автоматическая маршрутизация:** назначение объектов на техников по категории, локации и другим критериям
- **Эскалация:** повышение приоритета или переназначение при приближении дедлайнов
- **Приоритизация:** расчёт сроков по срочности, влиянию или инициатору
- **Синхронизация полей:** обновление связанных объектов при изменении исходного
- **Уведомления:** отправка предупреждений при выполнении условий

## 5. Workflow Items: Business Policies (Бизнес-политики)

Business Policies — специализированные правила автоматизации для тикетных объектов. Workflow Automation Engine проверяет условия через заданные интервалы.

### 5.1 Типы политик

| Тип | Назначение | Когда проверяется |
|-----|-----------|-------------------|
| **Ticket Escalation** | Изменение исполнителя, статуса, приоритета; уведомления | Непрерывно через заданные интервалы |
| **Ticket Prioritization** | Автоматический расчёт Due Date по важности | При создании или изменении |
| **Ticket Routing** | Автоматическое назначение тикета при выполнении условия | При создании или изменении |
| **Additional Ticket Automation** | Изменение свойств тикета при выполнении условия | При создании или изменении |

### 5.2 Условия политик

- **Для Escalation:** комбинация временных интервалов ("более X времени с даты" / "менее X времени до даты") и опциональных правил сопоставления значений
- **Для Routing:** фильтрация по Type, Category, Organization, Requester
- **Для Prioritization:** пересчёт Due Date при изменении приоритета или даты подачи
- **Для Additional Automation:** произвольные условия на основе значений полей

## 6. Workflow Components (Переиспользуемые строительные блоки)

Workflow Components — конфигурируемые блоки, используемые Actions и Triggers. Могут быть созданы внутри конкретного Action (одноразовые) или независимо для повторного использования.

### 6.1 Templates (Шаблоны)

- Назначают начальные значения полям при создании объекта
- Каждый объект должен иметь хотя бы один шаблон
- Значения шаблона переопределяют Default-значения полей
- Могут быть общими для нескольких действий или уникальными для одного

### 6.2 Forms (Формы)

- Настраиваемые диалоги или мастера для сбора пользовательского ввода
- Содержат **физические поля** (напрямую изменяют значения объекта) и **виртуальные поля** (временные, для передачи значений)
- Поддерживают фильтрацию полей, значения по умолчанию и инструкции
- Могут быть диалоговыми окнами или wizard-интерфейсами

### 6.3 Functions (Функции)

- **Переиспользуемые блоки операций**, вызываемые из Actions, Triggers и других Functions
- Принимают **параметры** (статические или определяемые в runtime)
- Поддерживают **именованные переменные** для хранения и передачи значений между операциями
- Обеспечивают **модульное проектирование** workflow — сложная логика инкапсулируется и переиспользуется
- Выполняют все те же операции, что доступны в Programming

### 6.4 Notifications (Уведомления)

| Тип | Описание |
|-----|----------|
| **E-mail Notification** | Шаблоны email-сообщений о событиях workflow |
| **Push Notification** | Текстовые сообщения на мобильные устройства техников |

Поддерживают статический и динамический контент (макросы, значения полей).

## 7. Programming: операции Workflow

Actions и Triggers содержат вкладку **Programming**, где определяются исполняемые операции. Это ядро программируемой бизнес-логики.

### 7.1 Доступные операции

| Операция | Описание |
|----------|----------|
| **Update Field** | Назначить значение полю объекта (статическое или вычисляемое) |
| **E-mail Notification** | Отправить email-уведомление |
| **Push Notification** | Отправить push-уведомление |
| **Execute Function** | Вызвать переиспользуемую функцию |
| **Create Object** | Создать новый связанный объект (с шаблоном, связями, управлением триггерами) |
| **Act on Related Object** | Выполнить операции над связанными объектами (с фильтрацией, поддержкой LOOP/END LOOP) |
| **Relate To** | Установить связь между текущим и другим объектом |
| **Unrelate** | Удалить связь между объектами |
| **External Command** | Выполнить PowerShell-команды и скрипты |
| **Execute SQL** | Выполнить SQL-выражения/скрипты |
| **Add Attachments** | Прикрепить файлы к объекту |
| **Message Box** | Показать pop-up диалог пользователю |
| **Abort** | Прервать выполнение Action/Trigger без сохранения |
| **Declare Variable** | Создать именованный контейнер данных |
| **Set Variable** | Назначить значение переменной |

### 7.2 Условная логика (IF/ELSE)

Операции могут быть обёрнуты в условные блоки:

```
IF (Criteria)
   [Операции 1]
ELSE IF (Criteria 2)
   [Операции 2]
ELSE
   [Операции 3]
END IF
```

- Поддерживаются вложенные условия
- Операции выполняются только если критерий оценивается как true
- Обеспечивает ветвящуюся логику внутри одного workflow item

### 7.3 Порядок выполнения

Операции выполняются в порядке списка с двумя исключениями:
1. **Email Notifications** выполняются последними (с пост-обработанными значениями placeholder)
2. **Execute SQL** выполняется после всех Update Field и Create Object

### 7.4 Act on Related Object — каскадные операции

Позволяет выполнять операции над связанными объектами:
- **Referenced objects** (прямые связи через атрибуты)
- **Neighboring objects** (непрямые связи)
- **Related objects** (через типы связей)
- Поддерживает фильтрацию
- Структура **LOOP/END LOOP** для обработки множества объектов

## 8. Система условий (Criteria Builder)

Условия используются повсюду в workflow: в триггерах, доступности действий, условных операциях.

### 8.1 Три типа условий

| Тип | Описание |
|-----|----------|
| **Standard Condition** | Сравнение значений полей или проверка состояния поля. Операторы: equals, is less than, is not blank, is unique, has been changed, is member of и т.д. |
| **Compare Two Date Fields** | Сравнение двух дат с операторами is before / is after. Поддержка расчёта разницы (by at least, by most, by exactly). Учитывает рабочий календарь |
| **SQL Criterion** | Произвольный SQL-скрипт WHERE, возвращающий TRUE/FALSE. Для сложной логики за пределами стандартных условий |

### 8.2 Логические операторы

Несколько условий объединяются операторами:
- **AND** (по умолчанию)
- **OR**
- **NOT AND**
- **NOT OR**

Условия группируются в группы для сложных выражений. Доступен предпросмотр перед применением.

## 9. Жизненный цикл объектов: статусы, стадии, переходы

### 9.1 Статусы

**Статусы** — пользовательские классификационные значения, определяющие фазы жизненного цикла объекта. Платформа предоставляет механизм полей и классификации; организация определяет конкретные значения и их значения.

Объекты могут находиться в:
- **Активных статусах** (Assigned, In Progress, Pending Approval...)
- **Неактивных статусах** (Closed, Retired, Inactive) — триггеры на таких объектах не срабатывают

### 9.2 Lifecycle Stages (Стадии жизненного цикла)

Стадии — высокоуровневая группировка статусов. Одна стадия может содержать несколько статусов, но статус принадлежит только одной стадии.

**Пример для Change Request (6 стадий по умолчанию):**

```
Submission → Approval → Planning → Implementation → Review → Close
(Assigned,   (Pending    (On Hold,  (In Progress,   (Tested,  (Completed,
 Draft)       Approval,   Approved)   Implemented)    Failed)   Close)
              Rejected)
```

**Визуализация:** стадии отображаются как графический индикатор прогресса в Web App и Self Service Portal.

**Объекты со стадиями:** Change Request, Incident, Problem, Service Request, Work Order, Purchase Order.

### 9.3 Переходы (Transitions)

Переход — перемещение объекта между стадиями жизненного цикла. Пользователь осуществляет переход через выполнение **Step Action** (например, Assigning, Implementing, Resolving).

Платформа **не имеет декларативной машины состояний** с явными определениями переходов. Вместо этого переходы **имплицитно закодированы** в условиях и операциях Step Actions:
- Условие Step Action определяет, при каких статусах оно доступно
- Операция Update Field в Programming меняет значение поля Status

### 9.4 Terminal Status (Терминальный статус)

Пользовательская характеристика классификационного значения, указывающая, что объект считается завершённым для целей автоматизации и управления. Терминальные статусы не предопределены платформой — они являются частью пользовательской бизнес-логики.

## 10. Workflow Parameters (Параметры рабочего процесса)

Центрально управляемые конфигурационные значения, которые ссылаются на них из множества workflow items через placeholder (WFP).

### 10.1 Назначение

| Возможность | Описание |
|-------------|----------|
| **Централизованная конфигурация** | Изменение значения в одном месте обновляет все ссылки |
| **Устранение дублирования** | Единое значение вместо копий в множестве мест |
| **Упрощение настройки** | Модификация поведения workflow без редактирования отдельных items |

### 10.2 Примеры использования

- Исполнитель по умолчанию для категории
- Порог тайм-аута эскалации
- Email получателя уведомлений
- Количество дней для автоматического закрытия завершённых инцидентов

### 10.3 Express-контекст

В Express, где предустановленный workflow не предполагает прямого редактирования, параметры конфигурации предоставляют контролируемую поверхность для настройки бизнес-логики (исполнители, пороги, настройки маршрутизации/эскалации) удобным для пользователя способом.

**Навигация:** `Workflow and Business Logic > Workflow Configuration`

**Правило:** присвоение значения **0 (ноль)** любому параметру **отключает** его.

## 11. Expression Builder (Построитель выражений)

Инструмент для построения динамических значений, используемый во всех компонентах workflow.

### 11.1 Области применения

- Назначение динамических значений полей
- Построение SQL-выражений
- Создание текстовых выражений
- Вставка placeholder (системные макросы, поля объекта, параметры, настройки заданий, поля email)

### 11.2 Типы макросов

| Тип | Описание |
|-----|----------|
| **System macros** | Дата/время, информация о пользователе |
| **Object fields** | Любое поле текущего или связанного объекта |
| **Workflow Parameters** | Параметры, передаваемые между операциями |

## 12. Processes (Процессы) — организационная группировка

Actions и Triggers могут быть **сгруппированы в процессы** (Processes) для организационных целей.

**Важно:** группировка по процессам используется **исключительно для классификации** — она не меняет поведение выполнения.

- Каждый процесс может включать workflow items из различных классов объектов
- Навигация: `Workflow and Business Logic > Advanced > Processes > Process List`
- Отображаются только процессы с назначенными Actions/Triggers

## 13. Automation Server (Сервер автоматизации)

Windows-сервис, выполняющий автоматизированные задачи независимо от основного приложения.

### 13.1 Категории заданий

| Категория | Описание |
|-----------|----------|
| **Outgoing E-mail** | Отправка исходящих email-уведомлений |
| **Alloy Discovery Sync** | Синхронизация с Alloy Discovery |
| **Active Directory Sync** | Синхронизация с AD |
| **Mail Connector** | Обработка входящей почты |
| **Archiving** | Архивирование и очистка данных |
| **Reporting** | Генерация отчётов по расписанию |
| **Workflow Triggers** | Проверка On Condition-триггеров |
| **Workflow Scheduled Tasks** | Выполнение запланированных workflow-задач |
| **Recurrent Tickets** | Генерация повторяющихся тикетов |

### 13.2 Характеристики

- Автоматическое обновление статуса заданий (каждые 10 секунд)
- Автоматическое завершение неотвечающих заданий (после 1 часа)
- Режим **High Availability** с primary/failover экземплярами
- Журналирование всех событий для аудита

## 14. PowerShell-интеграция

Workflow engine интегрирован с PowerShell через операцию **External Command**:

| Контекст | Описание |
|----------|----------|
| **Client-side** | Выполнение на стороне клиента (Desktop App) |
| **Server-side** | Выполнение на стороне сервера |
| **Embedded** | Встроенные PowerShell-выражения |

Контекст выполнения зависит от инициатора:
- **Desktop App:** поддерживает client и server
- **Web App, SSP, API, Automation Server:** только server

## 16. Модель взаимодействия компонентов

### 16.1 Flow создания объекта

```
Пользователь → Create Action (из меню New)
                  │
                  ├── Template → инициализация полей
                  ├── Form → сбор пользовательского ввода
                  └── Programming → выполнение операций
                        ├── Update Field (вычисляемые поля)
                        ├── Execute Function (переиспользуемая логика)
                        ├── Create Object (связанные объекты)
                        ├── Relate To (установка связей)
                        ├── Send Notification (email/push)
                        └── External Command (интеграции)
```

### 16.2 Flow жизненного цикла

```
Объект в активном статусе
     │
     ├── Step Action (доступен по статусу и роли)
     │     ├── Form → сбор ввода
     │     └── Programming
     │           ├── Update Field (Status → новый статус)
     │           ├── IF/ELSE (условная логика)
     │           ├── Act on Related Object (каскадные операции)
     │           └── Send Notification
     │
     ├── On Created/Modified Trigger (автоматически, ДО сохранения)
     │     └── Programming
     │           ├── Валидация данных
     │           ├── Вычисление полей
     │           ├── Abort (при ошибке валидации)
     │           └── Send Notification
     │
     └── On Condition Trigger (автоматически, по расписанию через Automation Server)
           └── Programming
                 ├── Эскалация (Update Field: Priority, Assignee)
                 ├── Уведомление о просроченных SLA
                 └── Автоматическое закрытие
```

### 16.3 Переиспользование через Functions

```
Action A ──┐
            ├──→ Execute Function "Notify Team" (parameters: team, severity)
Action B ──┤      ├── IF severity = "High"
            │      │     └── Send Email + Push
Trigger C ──┘      └── ELSE
                         └── Send Email only
```

## 17. Enterprise vs Express: различия модели бизнес-логики

### 17.1 Терминология

| Express | Enterprise | Тип связи |
|---------|-----------|-----------|
| Ticket | Work Order | Переименование (тот же класс) |
| Inventory Item | Hardware | Переименование |
| Business Rule | Trigger | Переименование |
| Business Policy | Trigger | Express-специфичная упрощённая форма |
| Library Item | Equipment | Переименование |
| Customization | Workflow and Business Logic | Переименование |

### 17.2 Сравнение возможностей

| Возможность | Enterprise | Express |
|-------------|-----------|---------|
| **Create Actions** | Полная Programming-логика | Нет. Заменены на Templates (только инициализация полей) |
| **Step Actions** | Создание и редактирование | Только Enable/Disable |
| **Triggers** | On Condition, On Created/Modified, On Delete (полная Programming) | Business Rules — ограниченный набор операций (Update Field, Function, Execute SQL, Send Notification) |
| **Business Policies** | N/A (используются Triggers) | Escalation, Prioritization, Routing, Additional Automation |
| **Functions** | Полное создание и редактирование | Нельзя создавать; можно вызывать встроенные |
| **Forms** | Полное создание | Нельзя создавать |
| **Validation Rules** | Доступны | Недоступны |
| **Lifecycle Stages** | Полная настройка | Для Tickets и Change Requests |
| **Workflow Parameters** | Полная система с пользовательскими параметрами | Доступны (Workflow Defaults) |
| **PowerShell** | Полная поддержка через External Command | Ограничена |
| **Conditional Logic** | IF/ELSE IF/ELSE/END IF блоки | В рамках условий Business Policies |

### 17.3 Классы настройки Express

| Класс | Уровень | Объекты | Возможности |
|-------|---------|---------|-------------|
| **Class 1** | Без настройки | Discovered Installations, Manufacturers | — |
| **Class 2** | Только справочники | Organization, Location, Vendor | Lookups |
| **Class 3** | Справочники + уведомления + шаблоны | Computer, Inventory Item, PO, Contract, License, KB, Person, Library Item | + Notifications, Templates |
| **Class 4** | Полная Express-настройка | Ticket, Change Request, Approval Request* | + Business Policies, Business Rules |

\* Approval Request: только Escalation Policy, без Business Rules.

### 17.4 Принцип Express

Express-администратор может управлять **какие** действия доступны, но не **что** эти действия делают. Бизнес-логика настраивается через параметры, политики и правила, но не через прямое программирование workflow.

## 19. Ключевые архитектурные характеристики модели

| Характеристика | Описание |
|----------------|----------|
| **Workflow-driven** | Поведение объектов определяется конфигурацией workflow, не кодом |
| **Имплицитные переходы** | Нет декларативной машины состояний; переходы закодированы в Step Actions |
| **Неинтрузивная автоматизация** | Business Policies и Triggers работают независимо от действий пользователя |
| **Трёхуровневая система условий** | Standard → Date Compare → SQL — от простых до сложных правил |
| **Каскадные операции** | Act on Related Object с LOOP для многообъектных workflow |
| **Модульная логика** | Functions обеспечивают переиспользование и единообразие |
| **Разделение данных и поведения** | Объект существует независимо от способа создания |
| **Аудит** | Automation Server журналирует все события для compliance |
| **Конфигурация-первый подход** | Бизнес-объекты, workflow, классификации и интеграции конфигурируются, не кодируются |
